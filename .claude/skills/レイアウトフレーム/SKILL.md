# web-layout-frame スキル（弊社オリジナル）

## 目的
ワイヤーフレームで確定した「レイアウト大枠」に対して、**実装に必要なレイアウト詳細値** を加え、
実装チーム向けの設計書として完成させるフェーズ。

弊社オリジナルの特色：
- **色・フォント・装飾は変更しない**
- **レイアウトのみに専念**：パディング・マージン・高さなど、実装時に必要な具体値を決定
- Figma上で「パディングはこのくらい、マージンはこのくらい」が一目でわかる設計書に

---

## プロセス（3段階）

### Phase 1: Figma でレイアウトフレーム作成
- web-wireframe を Figma で複製
- 各セクションのレイアウト詳細化：
  - **見出しのサイズ・行高・マージン** を決定
  - **テキストブロックの幅・パディング** を決定
  - **要素間のマージン**（px単位）を記入
  - **ビジュアル領域のサイズ・アスペクト比** を確定
  - **ボタン・フォーム要素のサイズ・パディング** を指定
- **フォント**：Noto Sans Japanese で全て統一
- **色**：グレースケール（ワイヤーフレームと同じ）
- 画像・アイコンプレースホルダーはそのまま残す

### Phase 2: やり取りで調整
ユーザーからのレイアウト指摘を受け、Figma で修正
- 「このセクションの上下マージンをもっと広くしたい」
- 「見出しと本文の間隔を調整」
- 「カードのパディングを変更」
など

修正後、再度 Figma でプレビュー確認。

### Phase 3: 確定 & 実装指示作成
- レイアウト詳細値が確定
- Figma から：
  - スクリーンショット（レイアウト確定版）
  - 要素ごとのサイズ・パディング・マージン指定一覧
- これらを実装チームに渡す
- → コーディング開始

---

## 出力物

### Figma ファイル
```
レイアウトフレーム.fig
├── 各セクション（レイアウト詳細化版）
│   ├── パディング指定（上下左右）
│   ├── マージン指定（要素間）
│   ├── 見出し・テキストのサイズ・高さ
│   ├── ビジュアル領域のサイズ
│   ├── ボタン・フォーム要素のサイズ・パディング
│   └── テキストコンテンツ（web-wireframe のまま）
└── レイアウト仕様書（別途）
```

### 実装指示書（Figma + テキスト）
```
要素ごとのレイアウト指定
- セクション上下マージン：80px
- セクション左右パディング：40px
- 見出しサイズ：36px
- 本文サイズ：16px
- 見出しと本文の間隔：20px
- など

コンポーネント仕様
- ボタン：padding: 16px 48px, height: 48px, など
- カード：padding: 24px, margin-bottom: 20px, など
```

### 特徴
- **色・フォント：ワイヤーフレームと同じ**（変更なし）
- **レイアウトのみ詳細化**：実装時に「どのくらいのスペースを確保するか」が分かる
- **実装者が直接参考にできるレベル**

---

## 次フェーズへのバトンタッチ

レイアウトフレーム確定後：
- 実装チーム（コーダー）に Figma + 実装指示書を渡す
- → HTML/CSS でコーディング開始
- → 動作・ブラウザ確認テスト
- → 本番環境へデプロイ

## 弊社オリジナルの特色

### フェーズの明確な分離
**web-wireframe（ワイヤーフレーム）**
- 何を、どこに置くか（配置の大枠）

**web-layout-frame（レイアウトフレーム）**
- パディング・マージン・高さ（配置の詳細化）
- 色・フォント・装飾は変更しない

### メリット
- **レイアウトのみに専念**：修正が早い
- **色・フォント固定**：ユーザーが「色どうしようか」で悩まない
- **実装指示が明確**：コーディングが効率的
- **各フェーズでユーザー確認**：修正コストが低い

### 通常フローとの比較
通常：
```
Figma デザイン → コーディング → テスト
```

弊社フロー：
```
要件整理
  ↓
コンテンツマップ（web-ia） ← ユーザー確認
  ↓
ワイヤーフレーム（web-wireframe） ← レイアウト確認
  ↓
レイアウトフレーム（web-layout-frame） ← レイアウト詳細化＆確認
  ↓
コーディング ← 指示が明確で実装が早い
```

各フェーズでユーザー確認が入るため、修正コストが低い。
実装段階でもめごとが少ない。
