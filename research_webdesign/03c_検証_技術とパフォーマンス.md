# 検証：技術とパフォーマンス

## 検証の目的

Webデザインは見た目だけでなく、「速く・快適に表示されるか」も品質の一部である。しかし技術的な話題は専門用語が多く、デザイナーにとって敬遠されがちな領域でもある。本レポートでは、コードを書かないデザイナーでも理解できるレベルで、パフォーマンスに関わる技術要件を整理する。デザインの段階で何を意識すれば、エンジニアとの協業がスムーズになり、最終的にユーザー体験の質を高められるのかを明らかにすることが目的である。

---

## 1. Core Web Vitals 2026年版

Core Web Vitalsとは、Googleがサイトのユーザー体験を測定するために定めた3つの指標である。検索順位にも影響するため、デザイナーも概要を把握しておく価値がある。

**LCP（Largest Contentful Paint）** は、ページで最も大きなコンテンツ（メインビジュアルや見出しなど）が表示されるまでの時間を測る。目標は2.5秒以内。ファーストビューに巨大な画像や未最適化の動画を配置すると、この数値が悪化する。デザイナーとしては、メインビジュアルのファイルサイズと表示優先度を意識することが重要だ。

**INP（Interaction to Next Paint）** は、ユーザーがボタンをクリックしたりフォームに入力したりした際、画面が反応するまでの遅延を測る指標で、以前使われていたFID（First Input Delay）に代わり2024年3月から正式採用された。目標は200ミリ秒以内。複雑なアニメーションや、大量のJavaScriptを必要とするインタラクションはINPを悪化させる。デザイン上の操作が「重い処理」を伴わないか、エンジニアと事前に確認することが望ましい。

**CLS（Cumulative Layout Shift）** は、ページ読み込み中にレイアウトがガタッとずれる現象を数値化したもの。目標は0.1以下。広告枠やWebフォント、遅延読み込みの画像がサイズ未指定のまま挿入されると発生しやすい。デザイナーは画像や埋め込み要素に対して、あらかじめ表示領域（アスペクト比）を明示する設計を心がけるとよい。

## 2. 画像最適化の実践

Webページの総容量のうち、画像が占める割合は平均して50%前後と言われる。画像の最適化はパフォーマンス改善で最も効果が大きい領域の一つだ。

### WebP・AVIF形式の使い分け

**WebP** はGoogleが開発した画像形式で、JPEGと比較して25〜35%程度ファイルサイズを削減できる。現在ほぼすべての主要ブラウザが対応しており、実質的な標準形式になりつつある。**AVIF** はさらに新しい形式で、WebPより20〜30%小さくできるケースもあるが、エンコード（変換処理）に時間がかかり、一部古いブラウザでは未対応の場合がある。実務では、AVIFを第一候補、WebPをフォールバック（代替）、JPEGを最終手段として用意する三段構えが理想だが、工数とのバランスでWebP＋JPEG対応が現実的な選択肢となることも多い。

### レスポンシブ画像の設計

同じ画像でも、スマートフォンとデスクトップでは必要な解像度が異なる。幅375pxのスマートフォンに幅1920pxの画像を配信するのは、通信量の無駄である。デザイナーは、ブレイクポイントごとに何種類の画像サイズが必要かをあらかじめ定義しておくと、エンジニアが適切に実装しやすい。一般的には、小（〜640px）・中（〜1024px）・大（〜1920px）の3段階を用意することが多い。

### 遅延読み込み（Lazy Loading）の注意点

遅延読み込みとは、画面に表示されるまで画像の読み込みを遅らせる技術である。ページの初期表示を高速化する効果がある。ただし、ファーストビュー（スクロールせずに見える範囲）の画像に遅延読み込みを適用すると、逆にLCPが悪化する。デザイナーは、どの画像がファーストビューに入るかを把握し、「この画像は即時読み込み、それ以外は遅延読み込み」という指示をデザインデータに添えると親切だ。

### 画像サイズの目安

一般的な目安として、1枚あたりの画像は**200KB以下**に収めることが望ましい。ヒーローイメージ（メインビジュアル）でも300〜500KB程度に抑えたい。ページ全体の画像合計は**1.5MB以内**を目標とすると、モバイル回線でも快適に閲覧できる。

## 3. フォント最適化

カスタムフォント（Webフォント）はデザインの個性を表現する強力な手段だが、パフォーマンスへの影響も大きい。特に日本語フォントは文字数が多く、英語フォントの数十倍のファイルサイズになることがある。

### Webフォントの読み込み戦略

Webフォントの読み込みには時間がかかる。特に日本語フォントは1書体あたり数MBに達することもあり、読み込み完了までテキストが表示されない、または別のフォントで表示された後に切り替わるという問題が発生する。対策として、Google Fontsなどのサービスでは必要な文字だけを動的に配信する仕組みが用意されている。デザイナーとしては、使用するフォントの種類を必要最小限に絞る（理想は2書体以内、ウェイトも3種類程度まで）ことが、最も効果的な最適化になる。

### FOUT・FOITの対策

**FOIT（Flash of Invisible Text）** は、Webフォントが読み込まれるまでテキストが見えなくなる現象。**FOUT（Flash of Unstyled Text）** は、最初にシステムフォントで表示され、読み込み完了後にWebフォントに切り替わる現象で、テキストがチラつくように見える。現在の主流はFOUT方式（まずシステムフォントで表示し、後からWebフォントに差し替え）で、ユーザーがコンテンツを素早く読める利点がある。デザイナーは、フォント切り替え時にレイアウトがずれないよう、システムフォントとWebフォントのサイズ感が近い組み合わせを選ぶことを意識するとよい。

### サブセット化による軽量化

サブセット化とは、フォントファイルからサイト内で実際に使用する文字だけを抽出し、ファイルサイズを削減する手法である。たとえば日本語フォントの場合、JIS第一水準の約3,000字に絞るだけでもファイルサイズを大幅に削減できる。さらに、見出し専用のフォントであれば、使用する文字を限定してサブセット化することで、数十KB程度まで軽量化できるケースもある。デザイナーは「このフォントは見出しだけに使う」「本文には使わない」といった使用範囲を明確にすることで、サブセット化の精度を高める手助けができる。

## 4. デザインとパフォーマンスのトレードオフ

デザインの理想とパフォーマンスの現実は、しばしば対立する。重要なのは「どちらかを諦める」ではなく、「どこで折り合いをつけるか」を意識的に判断することだ。

### アニメーション vs 表示速度

スクロール連動アニメーションやパララックス効果は印象的だが、大量のJavaScript処理を伴い、INPやページの応答性に影響する。特にモバイルデバイスでは処理能力の制限からカクつきが生じやすい。代替策として、CSSアニメーション（`transform`や`opacity`の変更）はGPUで処理されるため比較的軽量である。「この演出は本当にユーザー体験を向上させるか」を自問し、装飾的なアニメーションは控えめにする判断も重要だ。

### カスタムフォント vs システムフォント

カスタムフォントはブランドの個性を表現できるが、読み込みコストが発生する。一方、システムフォント（游ゴシック、ヒラギノ角ゴ、Noto Sansなど、ユーザーの端末にあらかじめ入っているフォント）は読み込み不要で即座に表示される。妥協点として、見出しにのみカスタムフォントを使い、本文はシステムフォントにする方法がある。近年はシステムフォントの品質も向上しており、デザイン性を保ちながらパフォーマンスを確保しやすくなっている。

### 高解像度画像 vs ファイルサイズ

Retinaディスプレイ向けに2倍・3倍の解像度画像を用意すると美しいが、ファイルサイズも比例して増大する。実際には、AVIF・WebP形式で適切に圧縮すれば、2倍解像度でも許容範囲のファイルサイズに収まることが多い。3倍解像度が必要なケースは限定的なので、通常は1倍＋2倍の2種類用意すれば十分である。

### 判断基準の考え方

トレードオフを判断する際は、以下の優先順位が参考になる。(1) ユーザーの目的達成に直結する要素は品質を優先する（商品画像、重要な操作ボタンなど）。(2) 装飾的な要素はパフォーマンスを優先する（背景アニメーション、装飾画像など）。(3) 迷った場合はモバイル環境を基準に判断する。日本のWebトラフィックの約7割はモバイルからのアクセスであり、モバイルでの快適さを確保することが、より多くのユーザーの体験を守ることにつながる。

## 5. デザイナー向けパフォーマンスチェックリスト

デザインデータをエンジニアに渡す前に、以下の項目を確認することでパフォーマンス問題を未然に防げる。

### 画像に関するチェック

- [ ] ファーストビューの画像は、最も重要な1枚に絞られているか
- [ ] 各画像のアスペクト比（縦横比）が明示されているか
- [ ] 装飾的な画像は、CSSで代替できないか検討したか
- [ ] 画像の書き出しサイズと形式をエンジニアに伝えているか

### フォントに関するチェック

- [ ] 使用フォントは2書体以内に収まっているか
- [ ] ウェイト（太さ）のバリエーションは3種類以内か
- [ ] 各フォントの使用箇所（見出しのみ、本文含む等）を明示しているか
- [ ] フォント読み込み前にシステムフォントで表示されても許容できるレイアウトか

### レイアウトに関するチェック

- [ ] 遅延読み込みされる要素の表示領域が確保されているか
- [ ] 広告やサードパーティ要素の配置場所とサイズが指定されているか
- [ ] スクロール時にレイアウトが大きくずれる設計になっていないか

### インタラクションに関するチェック

- [ ] アニメーションの種類と必要性を整理しているか
- [ ] モバイルで重い処理が発生するインタラクションがないか
- [ ] アニメーションをオフにした場合の代替表示を用意しているか（アクセシビリティ配慮）

### 確認ツール

デザイン段階でもパフォーマンスの概況を把握できるツールがある。**PageSpeed Insights**（Googleの無料ツール）にURLを入力するだけで、Core Web Vitalsのスコアと改善提案が得られる。競合サイトの分析にも活用でき、「このくらいの表示速度を目標にしたい」という基準づくりに役立つ。また、Chromeブラウザの開発者ツール内にある**Lighthouse**でも同様の計測が可能だ。
